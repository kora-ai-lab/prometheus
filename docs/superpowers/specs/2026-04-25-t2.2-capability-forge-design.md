# T2.2 — Capability Forge Design

**Date:** 2026-04-25
**Status:** Approved
**Parent:** T2.1 Capability Engine
**Next:** Implementation plan → `plans/2026-04-25-t2.2-capability-forge.md`

---

## 1. Overview

The Capability Forge creates new tools on-demand when no existing solution is found. It operates as a two-step LLM cycle: generate a **spec**, then generate **code** from that spec. Tests run automatically; up to 3 retry cycles on failure. Generated tools are stored in `~/.prometheus/capabilities/forged/` and integrate into the context manager's Block C so future sessions can use them.

---

## 2. Data Flow

```
User task (e.g., "parse FAO XML into JSON")
        ↓
Forge.Forge(ctx, task)
        ↓
┌─ Cycle (max 3×) ─────────────────────────┐
│  1. LLM generates spec (Spec struct)      │
│  2. Validate spec completeness            │
│  3. LLM generates code from spec         │
│  4. Write tool + test files              │
│  5. Run auto-tests (syntax + exec)       │
│  6. If pass → mark verified, break       │
│  7. If fail → collect errors, retry      │
└──────────────────────────────────────────┘
        ↓
Store in ~/.prometheus/capabilities/forged/{name}/
        ↓
Register in capability registry (forged type)
        ↓
Update Context Manager Block C (system prompt)
        ↓
ForgeResult {Name, Path, Verified, Language}
```

---

## 3. Spec Struct

```go
type Spec struct {
    Name        string   // tool name (kebab-case)
    Language    string   // "python" | "bash" | "go"
    Description string   // one-line purpose
    Inputs      []string // CLI args / stdin fields
    Outputs     []string// stdout fields / files produced
    Constraints []string// e.g., "offline-capable", "retry on network error"
    MainFile    string   // filename for main code (e.g., "fao_parser.py")
    TestFile    string   // filename for tests (e.g., "test_fao_parser.py")
}
```

Spec is generated via LLM call with task description as context.

---

## 4. Language Priority

Python → bash → Go

- Python: try first (most flexible, pip/npm compatible)
- bash: fallback for system utilities
- Go: last resort (requires `go run`, binary output)

---

## 5. Test Strategy

Three-level test per generated tool:

| Level | What | Pass criteria |
|-------|------|---------------|
| Syntax | `python -m py_compile` / `bash -n` | No syntax errors |
| Execution | Run the script with sample input | Exit code 0 |
| Functional | Run the test file (pytest, etc.) | All assertions pass |

Timeout per level: 30 seconds.

---

## 6. File Layout (per forged tool)

```
~/.prometheus/capabilities/forged/{name}/
├── meta.toml       # type = "forged", language, verified
├── {main_file}     # e.g., fao_parser.py
├── test_{name}.py  # unit tests
└── README.md       # auto-generated docs
```

**meta.toml format:**
```toml
type = "forged"
language = "python"
name = "fao-parser"
verified = true
forged_at = "2026-04-25T10:00:00Z"
retry_count = 0
```

---

## 7. Block C Update

After a successful forge, the context manager's Block C gets an entry appended:

```
## Forged Capabilities
- fao-parser: parse FAO XML into JSON | python | ~/.prometheus/capabilities/forged/fao-parser/
```

This is saved to the active context snapshot in SQLite so future sessions load it.

---

## 8. Error Handling

| Condition | Action |
|-----------|--------|
| LLM call fails | Return error, no forge retry |
| Empty/invalid spec | Immediate retry with error context |
| Test timeout (30s) | Count as failure, retry |
| All 3 cycles fail | Store tool as `verified=false`, return ForgeResult |
| Storage write fails | Return error |

---

## 9. Engine Integration

In `engine.go`, the `Ensure()` method will call `Forge.Forge()` when all lookup strategies fail:

```go
// In Ensure() after TryInternet:
if errors.Is(err, ErrCapabilityNotFound) {
    result, forgeErr := e.forge.Forge(ctx, task)
    if forgeErr != nil {
        return fmt.Errorf("forge failed: %w", forgeErr)
    }
    return nil // tool is now available
}
```

---

## 10. Testing

- **Unit tests:** Mock LLM responses, verify spec/code/test files written correctly
- **E2E:** Mock LLM with pre-recorded responses, verify tool executes
- **Integration (behind flag):** Real LLM call, feature-gated with `PROMETHEUS_FORGE_ENABLED=true`
